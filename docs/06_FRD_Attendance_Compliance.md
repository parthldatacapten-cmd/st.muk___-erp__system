## 1. Executive Summary

This document details the functional requirements for **Next-Gen Attendance Systems** (NFC & QR-based) and **Automated Compliance Reporting** (NAAC, NEP, AICTE). These features address critical pain points in Indian higher education where manual data collection for accreditation is time-consuming and error-prone.

## 2. Smart Attendance System

### 2.1 NFC-Based Attendance (Mobile-to-Mobile)
**Context:** Modern Android smartphones have built-in NFC readers/writers. This feature leverages student phones as "ID Cards" and teacher phones as "Scanners" (or vice versa), eliminating hardware costs.

**Functional Requirements:**
- **FR-NFC-01:** System shall generate a unique, encrypted **NFC Tag Emulation** payload for each student's profile, stored securely in their mobile app wallet.
- **FR-NFC-02:** Teacher's mobile app shall act as an **NFC Reader**. When a student taps their phone against the teacher's phone, attendance is captured.
- **FR-NFC-03:** System shall validate proximity: Attendance is only marked if the NFC handshake occurs within 4cm range.
- **FR-NFC-04:** **Anti-Spoofing:** The NFC payload shall include a time-sensitive OTP (valid for 30 seconds) generated by the server to prevent replay attacks.
- **FR-NFC-05:** Offline Mode: If the teacher's device is offline, attendance records are stored locally and synced when connectivity is restored (with timestamp verification).
- **FR-NFC-06:** Fallback Mechanism: If NFC hardware is absent on either device, the app automatically suggests QR Code mode.

**User Story:**
> *As a teacher in a crowded lecture hall, I want to quickly tap student phones with my device so that attendance is marked in 2 seconds per student without needing physical ID cards.*

### 2.2 QR Code Projector Attendance
**Context:** For large lecture halls where individual tapping is slow, the teacher displays a dynamic QR code on the projector/screen, and students scan it from their seats.

**Functional Requirements:**
- **FR-QR-01:** Teacher's dashboard shall display a **Dynamic QR Code** on the projector screen that refreshes every 10 seconds.
- **FR-QR-02:** Student app scanner shall decode the QR, extract the session token, and submit attendance with GPS coordinates.
- **FR-QR-03:** **Geo-Fencing:** System shall verify that the student's GPS location is within the classroom boundaries (defined by admin) before marking attendance.
- **FR-QR-04:** **Time-Window Validation:** QR codes expire after 2 minutes. Scanning an expired code results in "Late Mark" or rejection.
- **FR-QR-05:** **Proxy Prevention:** System shall detect if multiple students are scanning from identical GPS coordinates outside the classroom (indicating one person holding multiple phones).
- **FR-QR-06:** Visual Feedback: Student app shows "Attendance Marked ✅" with timestamp immediately upon successful scan.

### 2.3 Biometric & RFID Integration (Legacy Support)
**Functional Requirements:**
- **FR-BIO-01:** System shall integrate with existing biometric devices (eGate, Hikvision) via API or CSV import.
- **FR-BIO-02:** System shall support RFID card readers for institutions not ready for mobile-only solutions.

### 2.4 Attendance Analytics & Alerts
**Functional Requirements:**
- **FR-ATTY-01:** Real-time dashboard showing present/absent count per lecture.
- **FR-ATTY-02:** Automatic SMS/WhatsApp to parents if student is absent for >2 consecutive days.
- **FR-ATTY-03:** **Defaulter List:** Auto-generate list of students with <75% attendance (UGC mandate) at month-end.
- **FR-ATTY-04:** Condonation Request Workflow: Students can apply for attendance condonation with medical documents; approvers (HOD/Principal) can approve/reject digitally.

## 3. Automated Compliance Reporting (NAAC / NEP / AICTE)

**Problem Statement:** Currently, college staff manually compile Excel sheets for NAAC SSR (Self Study Report) and NEP compliance, taking weeks of effort. EduCore will automate this by pulling real-time data from the database.

### 3.1 NAAC (National Assessment and Accreditation Council) Module
**Focus:** Criteria I to VII automation.

**Functional Requirements:**
- **FR-NAAC-01:** **Criterion I (Curricular Aspects):** Auto-generate reports on syllabus completion %, number of new courses introduced, and feedback analysis from students.
- **FR-NAAC-02:** **Criterion II (Teaching-Learning):** 
  - Auto-calculate Student-Teacher Ratio.
  - Generate attendance trend reports (avg % per dept).
  - Extract pass percentages and result analysis over 5 years.
  - Compile list of students clearing competitive exams (from Assessment Module).
- **FR-NAAC-03:** **Criterion III (Research):** 
  - Track and list publications by faculty (stored in HR module).
  - Report on grants received and projects ongoing.
- **FR-NAAC-04:** **Criterion IV (Infrastructure):** 
  - Utilization reports of labs, library books issued, and computer hours logged.
- **FR-NAAC-05:** **Criterion V (Student Support):** 
  - List of scholarships disbursed, remedial classes conducted, and counseling sessions held.
- **FR-NAAC-06:** **Criterion VI (Governance):** 
  - Meeting logs of IQAC, grievances redressed (from complaint module).
- **FR-NAAC-07:** **One-Click SSR Export:** System shall generate the NAAC Self Study Report in the official PDF format with pre-filled tables and charts.
- **FR-NAAC-08:** **DVV Matrix:** Auto-fill the Data Validation and Verification matrix required for NAAC submission.

### 3.2 NEP (National Education Policy 2020) Compliance
**Focus:** Multidisciplinary education, credit tracking, and skill integration.

**Functional Requirements:**
- **FR-NEP-01:** **Academic Bank of Credits (ABC):** Integration readiness to push student credits to the ABC registry via API.
- **FR-NEP-02:** **Multiple Entry/Exit Tracking:** System shall track students exiting with Certificate (1 yr), Diploma (2 yr), or Degree (3/4 yr) and calculate eligible credits for each.
- **FR-NEP-03:** **Skill Courses Mapping:** Tag courses as "Skill-Based" vs "Theory" and generate reports on % of skill courses offered (NEP mandate: minimum 25%).
- **FR-NEP-04:** **Multidisciplinary Report:** Show data on students taking minor degrees/certificates outside their main department (e.g., Engineering student taking History minor).
- **FR-NEP-05:** **OBE (Outcome Based Education):** Map Course Outcomes (COs) to Program Outcomes (POs) and generate attainment levels automatically based on test scores.

### 3.3 AICTE & University Reports
**Functional Requirements:**
- **FR-AICTE-01:** Auto-generation of AIS (All India Survey on Higher Education) data points.
- **FR-AICTE-02:** Faculty qualification reports (NET/SET/PhD status) for AICTE approval.
- **FR-UNIV-01:** Affiliation renewal reports: Student strength, infrastructure details, fee utilization certificates.

### 3.4 Custom Report Builder
**Functional Requirements:**
- **FR-RPT-01:** Drag-and-drop report builder allowing admins to select fields (e.g., "Student Name", "Attendance %", "Internal Marks") and export to Excel/PDF.
- **FR-RPT-02:** Scheduled Reports: Auto-email monthly attendance summaries to HODs on the 1st of every month.
- **FR-RPT-03:** Audit Trail: Every generated report shall have a digital signature/log of who generated it and when.

## 4. Technical Architecture Notes

### NFC Implementation
- **Android:** Use `Host Card Emulation (HCE)` to emulate ISO 14443-4 tags.
- **iOS:** Limited background NFC; use active app scanning mode (Teacher opens app -> scans student phone).
- **Security:** Payload encrypted using AES-256; includes `UserID + Timestamp + HMAC_Signature`.

### QR Code Implementation
- **Library:** ZXing or Google ML Kit for fast scanning.
- **Dynamic Token:** QR contains `SessionID + TOTP (Time-based One-Time Password)`.
- **GPS Accuracy:** Require <30 meters accuracy for valid attendance.

### Reporting Engine
- **Data Warehouse:** Separate read-replica DB for analytics to avoid slowing down transactional systems.
- **Charting:** Use Chart.js or D3.js for visual dashboards.
- **Export:** PDF generation using Puppeteer or iText; Excel using Apache POI or SheetJS.

## 5. Acceptance Criteria

1. **NFC Speed:** Attendance must be marked in <2 seconds per tap.
2. **QR Security:** Expired QR codes must be rejected 100% of the time.
3. **Geo-Fencing:** False positives (marking attendance from home) must be <0.1%.
4. **NAAC Report Accuracy:** Auto-generated SSR data must match manual calculation within 1% variance.
5. **Load:** System must handle 500 concurrent QR scans in a 10-second window (large lecture hall scenario).

## 6. User Workflows

### Workflow A: NFC Attendance
1. Teacher opens "Take Attendance" -> Selects "NFC Mode".
2. Student unlocks phone and taps back of their phone to teacher's phone.
3. Teacher's phone beeps/vibrates; screen shows "Present: Rahul Sharma".
4. Repeat for next student.

### Workflow B: QR Attendance
1. Teacher clicks "Start Class" -> Projector displays large QR code.
2. Students open App -> Scan QR.
3. App validates GPS + Time -> Shows "Success".
4. Teacher dashboard updates real-time count: "45/60 Present".
5. After 2 mins, QR expires; teacher clicks "End Attendance".

### Workflow C: NAAC Report Generation
1. Admin navigates to "Compliance" -> "NAAC SSR".
2. Selects Academic Year (e.g., 2023-2024).
3. Clicks "Generate Report".
4. System aggregates data from Attendance, Exam, HR, and Library modules.
5. PDF downloaded with pre-formatted tables ready for submission.

## 7. Out of Scope (For Phase 1)
- Facial recognition attendance (privacy/legal complexities in India).
- Direct API submission to NAAC portal (manual upload still required by NAAC).
- Blockchain-based certificate issuance (planned for Phase 2).
